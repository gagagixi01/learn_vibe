Python 新人如何学习 Vibecoding：理解模式，再用 Claude Code CLI 做一个全栈小项目

下一页：指南 README

如果你已经学过一点 Python 语法，却总觉得自己“会写几行代码，但不会做项目”，那这篇文章就是写给你的。本文不是让你崇拜 AI，也不是让你把项目外包给 AI，而是带你学会一种新的学习方式：不是让 AI 替你写代码，而是让你学会用自然语言组织需求、拆分任务、验证结果，再和 AI 一起把项目做出来。

这篇文章分成两个阶段。第一阶段先把 vibecoding 讲清楚，让你知道它到底是什么、常见模式有哪些、为什么有些人越用 AI 越乱。第二阶段用一个足够小、但完整的 Python 全栈项目，带你通过 Claude Code CLI 在约 4 小时内走完 00-10 的核心练习，从需求到实现、从报错到复盘；如果你继续做第 11 节扩展复盘，整体时间会再往上加。

什么是 Vibecoding

vibecoding 可以简单理解为：你不只是“写代码”，而是在用自然语言驱动整个开发过程。你告诉 AI 你要做什么、为什么这样做、当前卡在哪、下一步希望得到什么；AI 再根据你的目标给出代码、解释、拆解方案、排错建议和重构意见。代码当然还是会出现，但它不再是唯一入口，需求表达和开发协作本身也变成了编程的一部分。

它和“把报错贴给 AI，然后复制答案”不是一回事。后者更像临时求助，通常能解决一两个局部问题，但很难帮你建立项目能力。vibecoding 更强调过程：你要先说明目标，再拆分任务，再观察结果，再决定下一步。换句话说，AI 不是答案仓库，而是副驾驶。

对 Python 新人来说，这种方式特别有价值。传统学习路径很容易让人困在语法细节里，学了函数、类、包管理、数据库连接，却始终没有做出一个完整系统。vibecoding 让你可以更早进入真实项目，把抽象知识放进具体场景里理解。不过它也有一个前提：你必须保持对项目的主导权。AI 可以加速，但不能代替理解。

本文也会参考 GitHub 上的一些官方项目和示例项目，比如 FastAPI 官方仓库、SQLModel 官方仓库、Claude Code 官方 README，以及少量 Todo/全栈模板项目。但这些参考的作用是帮助我们理解技术定位和项目结构，而不是让你去照抄一个大模板。对于第一次做练习的新人来说，可理解性比功能堆叠更重要。

第一阶段：理解 Vibecoding 的几种工作模式

很多人一提到 AI 编程，就默认只有一种方式：给一句需求，让它吐一堆代码。其实这只是最表层的一种。真正稳定的 vibecoding 往往是几种工作模式的组合，而不是一次性的大生成。

Prompt-Driven Coding：提示驱动式编码

这是最直觉的一种模式。你直接对 AI 说：“帮我写一个登录接口”“帮我做一个任务列表页面”“帮我修这个报错”，然后根据它的输出继续推进。它的优点非常明显：快、直接、进入门槛低。对新人来说，用它做探索、做 demo、试验某个功能特别合适。

但它的问题也同样明显。如果你没有边界，没有拆解，没有验证，它很容易从“高效”变成“失控”。AI 可能一次改很多文件，加入你没要求的逻辑，甚至把简单问题越修越复杂。所以，提示驱动式编码适合做起步，但不适合独自承担整个项目流程。

新人实操版：在本文里，你可以用它来生成第一版项目骨架、第一版需求草稿，或者让 AI 帮你解释某一步的最小实现，但不要直接让它“一次写完整个系统”。

SDD：Spec-Driven Development，规格驱动开发

SDD 的核心思想是：先写清楚我们要做什么，再让 AI 帮我们实现。这里的“规格”不一定是十几页正式文档，它可以只是一个很小的需求说明：系统给谁用、有哪些功能、哪些不做、成功标准是什么。

为什么这一步这么重要？因为 AI 最怕目标模糊。你只说“帮我做个任务系统”，它就会在脑子里调用无数个“任务系统”的公共模式：登录、权限、标签、分类、分页、搜索、通知、协作……结果很可能比你要的复杂得多。SDD 的意义，就是先收缩问题空间，把 AI 拉回你真正要的范围里。

对新人来说，SDD 还有一个更重要的价值：它逼着你在写代码之前先思考。你不再是机械地问“下一句代码怎么写”，而是先理解“这个系统到底要解决什么问题”。这一步会直接决定后面的工程质量。

新人实操版：在本文项目里，SDD 会落成一份很小的需求说明，明确这个任务管理系统做什么、不做什么，以及第一版只保留哪些功能。

TDD：Test-Driven Development，测试驱动开发

TDD 的经典顺序是：先写测试，确认测试失败，再写最小代码让测试通过。对一个初学者来说，完整、严格、重型的 TDD 可能会有点陡，但它背后的思想非常值得尽早建立：不要只看“代码写出来了没有”，要看“这个功能是否真的符合预期”。

在本文里，我们采用的是轻量 TDD。也就是说，你不一定要给每个页面和每一行逻辑都补齐复杂测试，但你至少要学会先定义行为，再做实现。比如写接口前，你先说清楚这个接口输入什么、返回什么、失败时应该怎样；写完以后再验证它是否真的满足这些要求。

轻量 TDD 对新人的好处，是帮助你形成“先验证再自信”的习惯。它会让你逐渐明白，能运行和正确实现是两件事。

新人实操版：在本文里，TDD 不会变成沉重的测试体系，而是变成每一步都先定义预期输出，再运行验证，例如“新增任务后页面必须出现这条任务”。

Iterative Development：迭代式开发

如果说 SDD 是先想清楚，Iterative Development 就是每次只做一小步。AI 协作最怕的问题之一，就是一次性目标太大。你让它“帮我做完整项目”，它往往会跨过很多中间验证点，最后你得到一堆你看不懂、也不敢动的代码。

迭代式开发要求我们把目标拆小。先做最小后端骨架，再做任务模型，再做获取列表接口，再做新增任务，再做页面展示。每一步都应该是可运行、可检查、可回退的。这样做有两个好处：一是你更容易理解当前在做什么，二是只要出错，定位范围也会很小。

这也是为什么真正会用 AI 的开发者，往往不是一句话让模型写完全部，而是通过很多轮小步前进，把复杂度分散掉。

新人实操版：你会把项目拆成需求、骨架、模型、接口、页面、调试、review 这些小步骤，每一步都能单独完成和验证。

Review-Driven Development：评审驱动开发

很多新人在 AI 协作里最容易忽视的一件事，就是“解释”和“评审”。代码跑起来并不等于你学会了。真正让你进步的，常常是这些追问：这段代码为什么这么写？这个文件负责什么？这是不是最简单的实现？有没有多余逻辑？还有没有更适合新人的写法？

所以，Review-Driven Development 的重点不是挑刺，而是把 AI 变成解释器和审稿人。你可以让它给你讲关键函数、给你做代码 review、给你指出命名问题、帮你补 README，甚至让它总结这一步到底学到了什么。对新人来说，复盘阶段的学习密度往往比“生成代码”那一步更高。

新人实操版：在本文最后一个学习单元里，你会把 Claude Code CLI 当成 reviewer，而不是 coder，让它帮你解释关键文件、指出明显风险、整理一份项目复盘。

新人最适合哪种组合

如果你是 Python 新人，我最推荐的不是单一模式，而是一种组合工作流：SDD + 小步迭代 + 轻量 TDD + Review。先写一个最小需求，确保目标明确；再把任务拆成足够小的步骤；每一步先定义预期，再写实现并验证；最后让 AI 帮你解释和审查。

这种组合的好处在于，它既不会把你困在繁重流程里，也不会让你滑向“全靠 AI 猜”的混乱状态。你会逐渐建立一种更可靠的工程节奏：先对齐、再实现、再验证、再解释。

参考项目说明：这篇文章借鉴了什么，没有借鉴什么

为了让这篇文章既靠谱又适合初学者，我参考了几类 GitHub 项目。第一类是官方能力来源，比如 fastapi/fastapi 和 anthropics/claude-code。它们帮助我们确认两个核心判断：FastAPI 适合新人入门，Claude Code 的定位是终端中的协作式 coding tool，而不是自动生成一切的黑盒。

第二类是结构参考，比如 fastapi/sqlmodel 和一些 Todo 类示例项目。这类仓库能帮助我们理解：一个最小的数据模型应该长什么样，一个任务系统最基本的 CRUD 链路如何组织。

第三类其实更像反例参考，比如 fastapi/full-stack-fastapi-template 这一类生产级模板。它们很强，但不适合第一次练习。因为它们往往同时引入前端框架、数据库服务、测试体系和部署配置，会让新人先陷入基础设施复杂度。

所以，本文不会让你 clone 一个大模板再照着改，而是让你参考优秀项目的思路，亲手做一个最小但完整的练习项目。

第二阶段开始前：为什么要用一个小型全栈项目来学

如果你只练单个 Python 文件，你当然可以学到函数、循环、异常处理这些基础知识，但你很难真正理解“做项目”意味着什么。真实开发至少要涉及四件事：用户看见什么、后端怎么处理、数据存在哪里、出了问题怎么排查。只有当你把这四件事串起来，你才会发现很多概念不再是分散的知识点，而是一个整体流程。

但与此同时，项目也绝不能太大。太大的项目会让初学者失去判断力：你不知道现在改的是哪一层，不知道报错来自哪里，也不知道 AI 输出的代码是不是合理。相比之下，一个小而完整的项目更适合作为第一次练习。它的功能少，链路却全，正好可以让你在可控复杂度里练会核心方法。

这也是本文不用重量级模板做第一练的原因。我们不否认大型模板的价值，但在第一轮学习里，最重要的不是“功能齐全”，而是“能亲手走完整条路径”。

我们要做什么项目

本文选择的练手项目是一个最小版的个人任务管理系统。它只有四个核心能力：查看任务列表、新增任务、标记任务完成、删除任务。没有用户系统，没有权限，没有复杂搜索，也没有炫目的前端效果。

之所以选这个项目，不是因为它“简单到没价值”，而是因为它恰好够完整。你会接触到数据库表设计、后端接口、模板页面、表单提交、状态更新和调试反馈。它能让你第一次感受到：一个全栈项目并不是很多神秘技术的堆叠，而是一条用户操作驱动数据变化的链路。

你也可以把它理解成一个教学载体。真正重要的不是这个任务系统本身，而是你在实现它时学会的开发方法。

项目技术栈设计

后端我们使用 FastAPI。原因很直接：它对 Python 初学者足够友好，语法清晰，文档完整，而且非常适合从接口的角度理解后端逻辑。官方对它的定位之一就是 easy to learn、fast to code，这与本文的目标高度一致。

数据库我们使用 SQLite。这样做的好处是省掉额外服务安装成本，你不需要先启动一个 PostgreSQL 容器或配置远程数据库，就能完成数据持久化。对于第一次练习来说，这能显著降低环境复杂度。

前端我们使用最简单的 HTML + Jinja2。这里的目的不是回避现代前端，而是为了把注意力集中在“前端如何把动作提交给后端、后端如何更新数据库、页面如何反映结果”这条主线上。等你理解了完整链路，再去学更复杂的前端框架会容易很多。

协作工具我们使用 Claude Code CLI。它的价值不在于“替你做完所有事”，而在于让你在终端里用自然语言持续推进开发：梳理需求、拆任务、生成最小代码、解释实现、分析报错、做 review。你不是在命令行里调用一个黑盒，而是在和一个能读懂上下文的开发搭档合作。

正式实战前：Claude Code CLI 是怎么参与开发的

在开始动手之前，我们要先给 Claude Code CLI 一个正确定位。它不是自动化建站器，也不是只会吐出代码片段的问答工具。你可以把它理解成终端里的协作式开发助手。你告诉它当前目录里有什么、你想完成哪一步、遇到了什么问题、希望它输出什么，它再基于这些上下文给出帮助。

对新人来说，Claude Code CLI 最适合做五类事情。第一，整理需求，把模糊想法变成更明确的功能范围。第二，拆任务，把“做个项目”变成一组能逐步完成的小步骤。第三，生成最小实现，尤其是骨架性代码。第四，解释代码和报错，帮你减少“运行了但没看懂”的状态。第五，做 review 和复盘，让你知道哪些地方其实还能更简单。做完一轮完整练习以后，你留下来的也不只是代码文件，还会有像 docs/spec.md、docs/tasks.md、docs/reflection.md、docs/extension-spec.md、docs/extension-review.md 这样的学习中间产物。

但它也有明确边界。它不适合在没有限制的情况下“一次性完成整个系统”，也不适合在你完全不给上下文时猜你的真实意图。你给它的目标越清晰，它越像一个靠谱搭档；目标越含糊，它越容易变成一个热心但跑偏的助手。

一份可以在约 4 小时内完成的核心学习安排

为了让这篇文章真正落地，给你安排一个约 4 小时的核心学习闭环。下面是一种可执行的学习节奏。熟悉 Python 和终端的读者可能接近 4 小时完成 00-10；如果你还想继续做第 11 节的小扩展和额外复盘，就再单独加时。

第 1 个学习单元用来建立认知。读完本文前半部分，理解 Prompt-Driven Coding、SDD、TDD、小步迭代和 Review 各自解决什么问题。产出物是：你能用自己的话写下这次项目要怎么做，不怎么做。

第 2 个学习单元用来准备环境。安装并确认 Claude Code CLI 可用，创建项目目录，准备 Python 虚拟环境，并安装这一轮正式项目依赖：fastapi、uvicorn、jinja2、pytest、httpx2、python-multipart。这里面有两个特别容易被忽略的包：httpx2 用来支撑 FastAPI TestClient 的接口测试，python-multipart 用来支撑后面 HTML 表单的提交解析。产出物是：虚拟环境可用，依赖安装完成，最小应用能启动。

第 3 和第 4 个学习单元用来做 SDD 和任务拆解。你先写出最小需求，再让 Claude Code CLI 帮你把任务拆成可执行步骤。这里的需求说明不只是业务目标，还会提前写出最小技术方案，例如后端使用 FastAPI、数据保存使用 SQLite、页面渲染使用 Jinja2。这样后面这些技术就不是“突然冒出来的新名词”，而是在规格里已经说明过职责的最小实现方案。产出物是：一份最小需求说明和一份按顺序排列的实战步骤清单。

第 5 和第 6 个学习单元用来完成后端与数据库的核心链路。你要搭好最小 FastAPI 应用，建立任务模型，实现查看和新增任务，再补上完成与删除的逻辑。到了第 7 个学习单元的 API TDD，你还会开始用 Claude Code CLI 在它自己的执行环境里创建 test_app.py、运行测试、记录一张“测试映射表”，把 list/create/toggle/delete 四个行为映射到真实 pytest node id，而不是死记教程里的测试函数名。产出物是：数据库里能看到任务记录，核心接口能完成 CRUD，而且你开始真正经历“先失败，再通过”的测试节奏。

第 7 个学习单元用来补页面和串联前后端。这个阶段的目标不是做漂亮页面，而是让你亲眼看到：用户点击按钮之后，后端收到了请求，数据库发生了变化，页面状态也跟着变了。这里会真正用到 python-multipart，因为浏览器表单提交给 FastAPI 时，需要它来解析 multipart/form-data 或其他表单数据。产出物是：浏览器里可以完整操作任务。

第 8 个学习单元专门留给调试、解释和复盘。因为第 8 节前半段已经把页面做成了可工作版本，所以第 9 节不会默认你“一定正好遇到报错”。如果当前项目已经正常新增、完成、删除任务，指南会让你用一个可控、可恢复的小错误来练调试，例如临时把新增任务表单的 action="/tasks" 改成 action="/"，制造一次 POST / 的 405 路径错误。然后再让 Claude Code CLI 先分析、再最小修复、最后恢复工作版本。产出物是：一份你自己能复述的项目结构说明，以及一份简短复盘。

实战项目第一步：先用 SDD 思路写需求

正式动手时，不要先问“帮我写代码”，而要先问“我们到底要做什么”。你可以先写一份非常小的需求说明，内容不需要很多，但必须清楚。比如：这个系统给个人使用；首页可以看到任务列表；用户可以新增任务、标记完成、删除任务；任务数据保存在本地数据库；这次练习不做登录、不做多人协作、不做复杂搜索。

然后把这份需求交给 Claude Code CLI，让它帮你整理成更清晰的版本。一个适合新人的提问方式可以是：请帮我为一个 Python 新人练习用的任务管理系统写一份最小需求文档，功能尽量简单，但要覆盖前端、后端和数据库，并明确哪些功能本次不做。

这一步的重点不是生成一份漂亮文档，而是把问题范围压缩清楚。只要需求稳定，后面很多 AI 输出都会更稳。这份文档通常会落成 docs/spec.md，里面除了写目标、不做什么、验收标准，也会用非代码方式提前写出最小技术方案：后面为什么会用 FastAPI、SQLite、Jinja2，以及它们分别负责什么。

这一步要产出什么：一份不超过十行的最小需求说明。

你可以这样问 Claude Code CLI：请根据这个任务系统目标，帮我整理一份适合 Python 新人的最小需求说明，并明确第一版不做什么。

完成后如何验证：你应该能清楚回答“这个项目做什么、先做哪些功能、哪些功能今天不做”。

实战项目第二步：让 Claude Code CLI 拆任务

有了需求以后，下一步不是直接写接口，而是先拆任务。你可以继续让 Claude Code CLI 帮你把项目拆成最小可执行步骤。比如：初始化项目；创建最小 FastAPI 应用；建立数据库模型；实现任务列表接口；实现新增任务接口；实现完成与删除操作；添加模板页面；串起表单和后端；最后做调试和 review。

这里有一个特别重要的标准：每一步都应该可验证。一个好任务，不是“完成后端逻辑”，而是“创建一个返回任务列表的接口，并在浏览器或文档页验证它能返回数据”。如果一个任务大到你不知道怎么验证，那它通常就还不够小。

你也可以把这个阶段看成是 vibecoding 的基本功训练。真正会用 AI 的人，不是更会下命令，而是更会切分工作。

这一步要产出什么：一个按顺序排列、每一步都可验证的任务清单。

你可以这样问 Claude Code CLI：请把这个项目拆成适合约 4 小时核心路线完成的最小步骤，每一步都写出完成后如何验证。

完成后如何验证：你能逐条说明“下一步要干什么，以及做完后看什么结果”。

实战项目第三步：搭建后端骨架

后端骨架的目标很简单：让项目先活起来。先创建项目目录，初始化虚拟环境，安装依赖，然后创建一个最小的 FastAPI 应用。第一版不要急着接数据库，也不要马上加很多页面。只要先有一个基础路由能跑起来，就已经非常好了。

你可以对 Claude Code CLI 说：请帮我生成一个适合 Python 新人的最小 FastAPI 项目结构，并解释每个文件的作用，只保留这次任务管理系统练习真正需要的内容。这样做的好处是，AI 不只是帮你搭骨架，还会顺手帮你建立项目结构感。

在这一阶段，请保持“最小实现”原则。如果只是为了验证应用能启动，那就不要提前加入多余配置。少一些复杂度，你就多一些理解空间。

这一步要产出什么：一个能启动的最小 FastAPI 应用。

你可以这样问 Claude Code CLI：请先只帮我搭一个最小 FastAPI 骨架，不要加入多余功能，并解释每个文件负责什么。

完成后如何验证：浏览器或本地访问时能看到首页响应，服务启动无报错。

实战项目第四步：加上数据库和任务模型

当后端骨架跑通以后，再引入数据库。这里推荐你先设计一个最小任务模型，只保留四个字段：id、title、is_done、created_at。这个模型足以支撑我们的练习，也足够让你理解数据是怎么从代码落到数据库里的。

此时你可以让 Claude Code CLI 先解释设计思路，再帮你写代码。比如问它：我只想做一个最小任务系统，请解释为什么这四个字段已经够用，然后再帮我生成对应的模型和建表代码。这样你拿到的不是一堆陌生定义，而是一套能解释得通的设计。

这一步还会帮助你建立一个非常关键的映射关系：页面上看到的是任务，后端里处理的是对象，数据库里存的是记录。三者其实是同一件事的不同视角。

这一步要产出什么：一个最小任务模型和可用的数据库表。

你可以这样问 Claude Code CLI：请先解释为什么 id、title、is_done、created_at 已经够用，再帮我生成最小模型代码。

完成后如何验证：数据库文件生成成功，插入一条测试任务后可以查询到。

实战项目第五步：实现核心接口

接下来开始实现最核心的四个能力：获取任务列表、新增任务、更新任务完成状态、删除任务。这里最好按照从简单到稍复杂的顺序推进。先做读取和新增，因为它们最容易形成正反馈；再做状态更新和删除，因为它们更能体现“数据变化”的概念。

在这一阶段，轻量 TDD 可以真正发挥作用。你不一定要先写完整测试文件，但至少要先说清楚每个接口的预期行为。比如：新增任务接口接收一个任务标题，成功后返回新增结果；如果标题为空，应该拒绝请求。然后再让 AI 根据这个预期帮你写最小实现。实际练习里，Claude Code CLI 会先创建 test_app.py，再在 Claude 内部运行 pytest，并把四个核心行为整理成一张“测试映射表”。所以你以后运行的往往不是教程里固定的测试函数名，而是自己项目里真实存在的 pytest node id。

每做完一个接口，就立刻验证一次。不要等四个接口都写完再测试。因为验证越晚，问题越难定位；验证越早，修复成本越低。这里还有一个心态很重要：通过标准不是“刚好看到 4 passed”，而是“核心行为测试都通过，额外测试如果存在也仍然在当前规格范围内”。

这一步要产出什么：四个最小但可用的核心接口，以及一组能映射到 list/create/toggle/delete 的真实测试。

你可以这样问 Claude Code CLI：先只实现新增任务接口，并基于这个预期来写代码：提交非空标题时成功创建，提交空标题时返回错误。不要改其他部分。

你也可以这样问 Claude Code CLI：请先列出“任务列表接口”最小需要满足的 3 个行为，再根据这些行为给出实现。

完成后如何验证：你能分别验证读取、新增、完成、删除四个动作，而不是只看最终页面是否凑巧能用。

实战项目第六步：补上最小前端页面

现在开始加页面。这里的目标不是做出炫酷 UI，而是让你第一次亲手看到完整链路：用户输入任务标题，浏览器把表单交给后端，后端写入 SQLite，再把最新任务列表返回给页面，页面因此显示新任务。

用 HTML + Jinja2 正好能让这个过程保持足够透明。你不会被复杂的前端工程配置分散注意力，而是可以专注在“页面怎么拿数据”“按钮怎么触发行为”“操作之后状态怎么刷新”这三件事上。

如果你之前一直把“前端”和“后端”想成两个完全不同的世界，这一步会帮你重新建立认知：它们其实只是同一条产品链路上的不同节点。

这一步要产出什么：一个可以操作任务的最小页面。

你可以这样问 Claude Code CLI：请给我一个适合初学者理解的最小 Jinja2 页面，只保留任务列表、新增表单和完成/删除按钮。

完成后如何验证：你能在浏览器里新增任务、刷新后仍能看到、点击按钮后状态会变化。

实战项目第七步：调试与修复

只要你开始把页面、后端和数据库真正串起来，报错几乎一定会出现。这不是坏事，反而是最宝贵的学习阶段。你会开始区分：有些错误来自环境，有些来自路由，有些来自字段名不一致，还有些来自模板和后端传值不匹配。

这时，把报错交给 Claude Code CLI 的方式也很重要。不要只说“帮我修一下”。更好的做法是：把完整报错贴给它，说明你刚做了什么操作，再要求它先解释最可能的原因，然后只给出最小修改方案。这样的对话更容易让你看懂问题，而不是被动接受一串改动。

如果它给出的修改太大，你要继续追问：这次只改和当前错误直接相关的部分，不要顺便重构其他逻辑。学会限制 AI 的改动范围，是 vibecoding 很重要的一项能力。

这里还有一个很适合新人的调试安排：如果第 8 节前半段已经把页面做通了，第 9 节也照样能练调试。做法不是等真实 bug 自己冒出来，而是先确认“新增、完成、删除”都能工作，然后人为制造一个受控错误，再把它修回去。默认练习是把新增任务表单的 action="/tasks" 临时改成 action="/"，这样点击新增时会走到 POST / 这条错误路径，常见结果是 405 Method Not Allowed。这个错误很小、可解释、可恢复，正适合新人练“收集证据 -> 让 Claude 先分析 -> 最小修复 -> 用同一动作重跑验证”。

这一步要产出什么：至少修掉一个你自己定位不了的真实问题，或者完成一轮受控调试练习，并能说清根因。

你可以这样问 Claude Code CLI：这是完整报错和我刚刚的操作，请先分析最可能的原因，只给我最小修复方案，不要顺便改动其他文件。

完成后如何验证：同样的操作再次执行时，错误不再出现，而且你能解释为什么它被修好了。如果你练的是受控错误，还要确认你已经把项目恢复回第 8 节的工作版本。

第九步以后：把方法继续用到一个小扩展

到这里，00-10 的核心路线其实已经完成了。接下来的 11 是一段可选延伸：它不是只让你写一个 mini spec 就停下，而是给你一轮完整的小扩展闭环。它发生在核心 4 小时之外，更像课后加练。

这轮扩展的起点通常是 docs/extension-spec.md。你会先收缩一个很小的扩展目标，例如“任务优先级”的第一条行为：如果创建任务时没有指定 priority，系统默认保存为 normal。然后你继续让 Claude Code CLI 帮你写第一条测试，推荐单独放在 test_extension.py，避免污染前面的核心 API 测试。接着先看到失败，再做最小实现，再用同一条测试和一次手动验证确认行为成立，最后写出 docs/extension-review.md。

这意味着第 11 节真正练的是：你能不能自己主导一轮小扩展，而不是永远跟着现成教程走。它仍然坚持同一套原则：只做一个方向、不同时加多个功能、测试失败时只修当前行为、review 要写清楚目标、改动、验证和下一步。

实战项目第八步：让 AI 帮你做代码解释和 review

当项目已经基本跑通，不要急着结束。这个时候最值得做的，是把 AI 从“实现者”切换成“解释者”和“评审者”。你可以让 Claude Code CLI 用初学者能懂的语言，解释 main.py 负责什么、数据模型文件负责什么、模板页面是怎么和后端连起来的。

接着，再让它做一次轻量 Review。让它检查命名是否清晰、逻辑是否重复、有没有对新人不友好的写法、README 里还缺什么说明。你甚至可以让它帮你补一个非常简单的项目总结，记录这次练习的目标、功能范围、项目结构和运行方式。

如果说前面的步骤让你把项目做出来，那么这一步才真正让你把项目“学进去”。

这一步要产出什么：一份你自己能复述的项目结构说明和一个简短 review 结果。

你可以这样问 Claude Code CLI：请用 Python 新人能听懂的方式解释这个项目里最关键的 4 个文件，并给我一个 5 条以内的轻量 review 清单。

完成后如何验证：你能不看代码也说出“请求从页面到数据库再回到页面”的路径，并能指出自己项目里至少一个可以改进的点。

这一套流程里，Python 新人到底学到了什么

做完这 4 小时核心练习以后，你收获的绝不只是一个任务管理系统。你学会的是另一种学习和开发方式。你开始知道如何从需求出发，而不是从代码片段出发；知道如何把一个大问题拆成可执行的小步骤；知道如何在每一步都要求可验证的结果，而不是盲目相信“看起来差不多”。

你还会第一次真正理解一个全栈项目的最小链路：页面如何发起操作，后端如何处理逻辑，数据库如何保存状态，错误信息又如何反过来指导你修改实现。更重要的是，你会开始习惯和 AI 进行工程化协作，而不是零散求助。

这套能力会远比“记住了多少 API”更有价值。因为它决定了你以后面对陌生项目时，能不能自己组织学习路径。

新人最容易踩的坑

第一个常见坑，是一开始就让 AI “帮我做完整系统”。这种提问看起来省事，结果往往是最难学会东西。因为系统一旦一次性展开，你就失去了对复杂度的控制。

第二个坑，是不写需求，边想边改。这样做的后果不是更灵活，而是 AI 会不断根据你临时变动的目标重新猜测方向，项目结构也会越来越乱。

第三个坑，是一次改太多地方。你也许只是想修一个表单提交问题，结果顺手把路由、模型、模板都动了。最后只要程序坏掉，你几乎不可能快速定位根因。

第四个坑，是只看“能不能跑”，不看“为什么这么实现”。如果你只是机械复制，那你很难把这次练习迁移到下一次项目里。

给 Python 新人的 Vibecoding 学习路线

如果你准备把这篇文章真正用起来，我建议你按三个层次继续练。第一层是复现层。先照着本文做完这个最小任务管理系统，不求快，重点是亲手走完整条链路。

第二层是改造层。项目跑通以后，给它加一个小功能，比如任务优先级、截止日期、筛选状态或者关键字搜索。这里的重点不是功能本身，而是你开始自己主导需求变化，再让 AI 协助实现。一个适合新人反复练习的小闭环是：docs/extension-spec.md -> test_extension.py -> 最小实现 -> docs/extension-review.md。

第三层是主导层。到这一步，你可以尝试自己先写需求说明、自己先拆步骤，再把更清晰的上下文交给 Claude Code CLI。这时你会发现，AI 仍然很有用，但你已经不再依赖它替你做决定，而是在用它放大自己的思考效率。

从长期来看，真正的目标不是更依赖 AI，而是更会主导 AI。只要你能定义目标、约束范围、验证结果、解释实现，你就已经在从“会用工具的人”变成“会做项目的人”。

结语

vibecoding 不是偷懒，也不是神话。它更像是一种新的开发协作方式，让新人可以更早接触真实项目、更快获得反馈、更系统地建立工程直觉。它最大的价值，不是让你少思考，而是逼着你用更清晰的方式思考。

如果你愿意把这篇文章变成一次真实练习，请不要只停留在阅读。先给自己留出约 4 小时，打开 Claude Code CLI，从最小需求开始，小步验证，再认真复盘；如果你还想继续做第 11 节扩展，再额外加一轮时间。只要你亲手走完这条路，你得到的就不只是一个任务管理系统，而是一套以后还能继续复用的学习方法。
