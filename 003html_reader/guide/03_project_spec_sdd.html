03. 用 SDD 写出最小项目规格

Source: guide/03_project_spec_sdd.md
上一页：02 创建正式项目目录并准备 Python 环境 下一页：04 把规格拆成可执行任务

预计用时

15-20 分钟。

本节位置

约 4 小时核心路线（00-10）的第 3 个学习单元。02 已经完成正式项目目录、虚拟环境、依赖和最小 main.py。本节不继续写代码，也不拆任务，而是先把项目规格写清楚。任务拆分属于 04，后端、数据库和页面实现属于后面的章节。

SDD 的意思是 Spec-Driven Development，也就是“规格驱动开发”。对 Python 新人来说，它最重要的价值不是写文档，而是先把“我要做什么”和“我暂时不做什么”说清楚。AI 很擅长补全缺失信息，如果你只说“做一个任务管理系统”，它可能主动加入登录、权限、搜索、部署、React、Docker 等内容。那些功能不一定错，但会让约 4 小时的核心练习失控。

这里还有一个常见困惑：后面章节会陆续出现 FastAPI、SQLite、Jinja2，但如果 docs/spec.md 完全不提它们，学生到了 05、06、08 就容易觉得这些技术像是突然冒出来的。本节依然不写任何实现代码，不过规格里可以提前说明“这个最小项目计划用什么技术，以及每种技术负责什么”。这样后面的实现章节就不是凭空加内容，而是在落实你已经写下的最小方案。

工具使用规则

终端用于确认当前目录是 task-vibe/、激活 task-vibe/.venv、创建 docs/、查看和检查 docs/spec.md。Claude Code CLI 用于把模糊项目想法整理成最小 SDD 规格，但学生必须阅读、删减和确认边界。不要让 Claude 在本节拆任务、写代码、创建接口、连接数据库或生成页面。

学习目标

完成本节后，你应该能做到 4 件事：

  1. 解释为什么正式写代码前要先写 docs/spec.md。
  2. 用 Claude Code CLI 生成一份范围很小的项目规格。
  3. 检查规格是否包含目标、不做什么、用户操作、数据字段、最小技术方案、页面与接口边界、验收标准。
  4. 明确知道本节只产出规格，不产出 docs/tasks.md，也不写任何新代码。

本节的核心不是“让规格看起来很专业”，而是让它能服务后续开发。好的规格应该让学生看完后知道：这个系统给谁用，最小功能是什么，哪些功能今天不做，用户会怎样操作，数据至少需要哪些字段，后面为什么会用到 FastAPI、SQLite、Jinja2，以及什么结果才算成功。

本节产出物

task-vibe/docs/spec.md

这份文件是后续课程的共同依据。04 会根据它拆任务，05-08 会根据它实现后端、数据库、API 和页面，10 会根据它做 Review。也就是说，规格写得越清楚，后面越不容易被 AI 带偏。

操作步骤

第 1 步：在终端确认目录和环境

本节所有操作都发生在正式项目目录 task-vibe/ 内。先在终端执行：

cd task-vibe
pwd
source .venv/bin/activate

pwd 的输出应该以 task-vibe 结尾。

第 2 步：在终端创建 docs 目录

规格文件放在 docs/ 里：

mkdir -p docs

这一步只创建文档目录，不创建代码目录，不创建数据库文件。

第 3 步：在 Claude Code CLI 中生成 SDD 规格

现在进入 Claude Code CLI，使用本节提示词。注意：这里让 Claude 做的是“写规格”，不是“写代码”。如果 Claude 开始输出任务清单、代码文件、数据库结构实现，说明它越界了，需要要求它收回到 docs/spec.md。

claude

Claude Code 提示词示例：

我们现在在 task-vibe/ 正式项目目录里,本次执行只做 SDD 规格，不拆任务、不写代码、不创建接口、不连接数据库、不创建页面。

请根据“个人任务管理系统”这个练习目标，创建或更新 docs/spec.md，写一份最小项目规格。要求：
1. 只做查看任务、新增任务、切换完成状态、删除任务；
2. 明确写出不做登录、权限、搜索、分页、部署、Docker、React；
3. 包含项目目标、用户范围、最小功能、不做什么、用户操作流程、数据字段、技术栈与职责、页面与接口边界、验收标准；
4. 验收标准必须是可以实际验证的动作；
5. 在技术栈与职责里，只说明为什么后面要用 FastAPI、SQLite、Jinja2，以及它们各自负责什么，不要写安装命令、代码片段或配置细节；
6. 只写 docs/spec.md 需要的 Markdown 规格，不输出任务清单，不输出代码。

如果 Claude 输出了代码或任务清单，继续在 Claude Code CLI 中输入：

请收回到 SDD 规格。本节不要拆任务，不要写代码，只保留 docs/spec.md 需要的项目规格内容。

第 4 步：回到终端检查 spec.md

完成后检查：

cat docs/spec.md

检查时不要只看有没有文件，要看内容是否真的清楚。重点看三块：不做什么 是否明确，技术栈与职责 是否解释了后续章节为什么会用这些技术，验收标准 是否可验证。如果 docs/spec.md 还没有形成稳定结构，让 Claude Code 按本节边界继续收缩。下面这个 Markdown 模板只用来对照结构，不要求完全一致：

参考修改示例：

# 最小任务管理系统规格

## 项目目标

做一个给个人使用的任务管理系统，用来查看、新增、标记完成和删除任务。

## 用户范围

只给单个本地用户练习使用，不做登录，不做多人协作。

## 最小功能

- 查看任务列表
- 新增任务
- 标记任务完成或未完成
- 删除任务

## 不做什么

- 不做登录、注册、权限
- 不做搜索、筛选、分页
- 不做部署
- 不做 React 或复杂前端框架
- 不做 Docker

## 用户操作流程

- 打开首页，看到任务列表
- 在输入框填写任务标题，提交后看到新任务
- 点击完成按钮，任务状态发生变化
- 点击删除按钮，任务从列表中消失

## 数据字段

- id：任务唯一编号
- title：任务标题
- is_done：是否完成
- created_at：创建时间

## 技术栈与职责

- FastAPI：负责提供最小后端和任务相关路由
- SQLite：负责把任务数据保存在本地，刷新页面后仍然能读到
- Jinja2：负责渲染最小 HTML 页面，让用户能直接在页面上操作任务

## 页面与接口边界

- 页面只需要一个首页
- 后端只处理任务列表、新增、切换状态、删除
- 数据保存到本地 SQLite

## 规格示例中的验收标准

- 首页能显示当前任务列表
- 输入非空标题后能新增任务
- 点击完成按钮后，任务完成状态能变化
- 点击删除按钮后，任务不再出现在列表里
- 刷新页面后，任务数据仍然存在

这个模板看起来简单，但已经足够支撑后面的全栈练习。

第 5 步：删掉范围外内容

如果规格里出现下面内容，本节要删掉：

  • 登录、注册、用户权限
  • 多人协作
  • 搜索、筛选、分页
  • React、Vue、复杂前端工程
  • Docker、部署、云服务
  • PostgreSQL、Redis
  • 自动生成 docs/tasks.md
  • FastAPI 路由代码、SQLite 初始化代码、Jinja2 模板代码、HTML 页面实现代码

这些功能以后都可以学，但不是本课程的最小版本。SDD 不是把想法写得越大越好，而是把今天要完成的范围收得足够清楚。

第 6 步：评审规格是否能指导后续学习

最后，请用学习者视角评审一次 docs/spec.md。评审不是让文档变复杂，而是确认它能帮助后面的章节继续推进。

第一，看它是否能回答“系统给谁用”。本项目只给单个本地用户练习使用，所以不需要用户表、登录页、权限模型。 第二，看它是否能回答“最小功能是什么”。本项目只需要查看、新增、切换完成状态、删除任务。第三，看它是否能回答“怎样证明做对了”。如果验收标准写成“任务系统可以正常使用”，这句话太空；如果写成“输入非空标题后，页面刷新后能看到新任务”，就可以验证。

你还要检查规格能不能交给 04 拆任务使用。好的规格会给 04 留出清楚的拆分依据，比如页面、数据字段、技术栈职责、用户操作、验收标准。坏的规格会让 04 不知道从哪里开始，只能让 Claude 自己猜。

可验证验收标准示例

验收标准要像“可观察结果”，不要像“主观愿望”。下面是对比：

模糊写法 更好的写法
系统要简单好用 首页能显示任务列表，列表中能看到任务标题和完成状态
新增功能要正常 输入非空标题并提交后，新任务出现在列表中
删除要没问题 点击删除后，该任务不再出现在列表中，刷新后仍然不存在
数据要保存 新增任务后刷新页面，任务仍然显示

这张表不用写进最终规格也可以，但你应该用它检查自己的 docs/spec.md。如果规格里的验收标准不能被 curl、浏览器操作或数据库查询验证，就说明它还不够清楚。

输入输出对照

输入 预期输出 学生要比较什么
个人任务管理系统，只做查看、新增、完成、删除任务。 docs/spec.md，包含目标、不做事项、用户操作、数据字段、技术栈与职责、页面与接口边界、验收标准 是否把模糊想法压缩成清晰边界。
Claude 输出的规格 一份不包含任务拆分和代码实现的 Markdown 文档 是否没有提前进入 04 或后续实现章节。
cat docs/spec.md 能看到完整规格内容 是否回到终端检查了 Claude 的产物。

验收标准

运行或检查：

cat docs/spec.md

通过标准：

  • docs/spec.md 已创建，并且只是一份项目规格。
  • 规格包含 5-8 个必要部分，功能范围保持最小。
  • 不做什么 明确排除了登录、权限、搜索、部署、Docker、React。
  • 规格能解释后面为什么会用 FastAPI、SQLite、Jinja2，但解释停留在职责层，不进入实现。
  • 验收标准可以实际验证，不是“体验良好”“功能完整”这种口号。

需要停下处理：

  • 如果规格变成产品愿景，写得很宏大但无法指导实现，先收缩到最小版本。
  • 如果规格变成任务清单，提前抢了 04 的内容，先删掉任务拆分。
  • 如果规格里没有解释后续技术为什么存在，补上职责说明后再继续。
  • 如果规格里出现 FastAPI 路由代码、数据库代码、Jinja2 模板代码或页面代码，不要进入下一节；让 Claude 收缩范围，只保留最小规格。

卡点与过关

常见卡点：

  • 把 SDD 当成写作文，写很多愿景，却没有验收标准。
  • “不做什么”写得太少，导致 Claude 后面主动加功能。
  • 完全不提后续最小技术方案，导致到了 05、06、08 才疑惑为什么要用这些技术。
  • 验收标准不可验证，例如“系统要好用”。
  • 让 Claude 顺手生成任务拆分，导致 03 和 04 混在一起。
  • 看到 Claude 输出代码就直接接受，忘记本节只写规格。

过关前确认：

  • 我已经创建了 docs/spec.md。
  • 我已经用终端检查过 docs/spec.md 内容。
  • 规格里明确写了目标、不做事项、用户操作、数据字段、技术栈与职责、页面与接口边界、验收标准。
  • 规格没有任务拆分，没有代码实现。
  • 我知道 04 才会把这份规格拆成 docs/tasks.md。
