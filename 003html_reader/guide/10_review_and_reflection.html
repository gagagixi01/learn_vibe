10. Review 项目，并把它讲清楚

Source: guide/10_review_and_reflection.md
上一页：09 出现错误时，怎样和 Claude Code CLI 一起调试 下一页：11 做一个小扩展，把方法完整用一遍

预计用时

20-25 分钟。

本节位置

约 4 小时核心路线（00-10）的第 8 个学习单元前半段。把项目从“能跑”复盘到“能讲清楚”。

工具使用规则

终端用于查看项目文件、必要时确认服务仍可运行。Claude Code CLI 用于解释关键文件、数据流和做轻量 Review。

学习目标

  • 用 Review 和复盘把项目从“做出来”变成“学进去”。
  • 学会向 Claude Code CLI 请求解释代码、解释数据流和做轻量 review。
  • 用自己的话说明关键文件分别负责什么、一次请求怎样流动、数据库什么时候被读写。
  • 区分“我完成了项目”和“我能解释它为什么这样工作”。
  • 输出一份自己的项目复盘文档，而不是复制 Claude 的总结。

本节产出物

  • 当前工作目录：task-vibe/
  • 新增文件：task-vibe/docs/reflection.md
  • 一份你自己能复述的项目结构说明

重点看复盘是否解释了项目结构、数据流和改进点，而不是只写“今天学到了很多”。

操作步骤

  1. 确认你还在 task-vibe/ 目录内，项目已经能正常运行。

Review 本身主要是阅读、解释和写复盘，不需要一直激活虚拟环境。如果服务已经停了，而你需要重新运行项目或测试，请先回到项目目录并激活正式项目环境：

cd task-vibe
source .venv/bin/activate
python -m uvicorn main:app --reload

如果只是查看文件、写 docs/reflection.md 或向 Claude 提问解释代码，不需要为了这些动作单独激活 .venv。Review 阶段不是重新加功能，而是检查自己是否真的理解已经完成的这个小应用。

  1. 先自己做一轮快速盘点。不要急着问 Claude，先在纸上或临时笔记里写 5 行：
  • 哪个文件是应用入口？
  • 哪个文件负责页面显示？
  • 哪个文件或哪段代码负责接收表单提交？
  • 数据保存在哪里？
  • 完成、删除、更新任务时，数据大概经历了哪些步骤？

这一步不要求完全正确。它的价值是暴露你现在真正理解了什么、哪里还只是“跟着做过”。

  1. 让 Claude Code CLI 用初学者能听懂的语言解释关键文件。

Claude Code 提示词示例：

请用 Python 新人能理解的语言解释这个项目的关键文件、请求流程和数据库流程。然后给我一份 5 条以内的轻量 review 清单，重点看：命名是否清楚、结构是否过重、是否有明显的初学者风险。
  1. 如果解释里有听不懂的地方，继续追问，但问题要围绕“我怎样自己讲清楚”。在你还没有先说出自己的理解之前，不要让 Claude 直接代写最终复盘。可以继续输入：
请不要替我写最终复盘。请把刚才的解释拆成 4 个问题来考我：关键文件、请求流程、数据库读写、一个设计取舍。每个问题后面给一个新人友好的参考答案，方便我对照自己是否讲清楚。

你可以让 Claude 解释、举例、提问、指出风险；但 docs/reflection.md 里最后留下来的话，要改成你自己的表达。判断标准很简单：如果老师指着其中一句问“这是什么意思”，你能不用 Claude 再解释一遍。

  1. 按下面的顺序做自我 review。每一项都先自己说，再看 Claude 的解释是否能帮你补洞：
  • 关键文件：说明 main.py、templates/index.html 和 SQLite 数据库分别负责什么。不要只写“负责后端”“负责前端”，要写出它们在任务列表应用里的具体作用。
  • 请求流程：从用户在浏览器里提交表单开始，讲到 FastAPI 路由接收请求、执行业务逻辑、返回或重新渲染页面。
  • 数据库流程：说明什么时候新增任务、什么时候查询任务列表、什么时候更新任务状态、什么时候删除任务。
  • 设计取舍：说明这个小项目为什么可以把逻辑放得比较简单，以及以后项目变大时可能需要怎样拆分。
  • 风险和改进：指出 2 到 3 个你现在看得到的风险，例如命名不够清楚、数据库操作集中在一起、错误提示还不够友好。

  1. 区分两种完成状态。第一种是“我 finished it”：页面能打开，按钮能点，任务能保存。第二种是“我 can explain it”：我能说明每个关键文件的职责，能按顺序讲出数据如何从页面进入后端、写入 SQLite、再显示回页面，也能说出当前实现的取舍和风险。本节真正要完成的是第二种。

  2. 根据这些说明，让 Claude Code 用你的回答创建 docs/reflection.md 草稿。草稿只能帮你组织语言，最后留下来的句子必须是你能解释清楚的句子。

Claude Code 提示词示例：

请根据我刚才对关键文件、请求流程、数据库流程、设计取舍、风险和改进的回答，创建 docs/reflection.md。

要求：
1. 使用我自己的理解来组织，不要写成正式项目报告；
2. 包含关键文件、请求流程、数据库流程、我学到的 3 件事、一个设计取舍、我看到的 2 个风险、下次想改进的 2 件事；
3. 如果我的回答不清楚，请先指出需要补充的问题，不要编造；
4. 写完后提醒我逐句检查，删掉我不能解释的话。

下面的结构只用来对照，不要求完全一致，也不要直接复制 Claude 的总结：

参考修改示例：

# 项目复盘

## 关键文件

- main.py：负责创建 FastAPI 应用、定义路由、接收表单请求，并调用数据库相关逻辑。
- templates/index.html：负责把任务列表显示成页面，也负责提供新增、完成、删除任务的表单或按钮。
- SQLite 数据库：负责把任务保存下来，让刷新页面之后仍然能看到任务状态。

## 请求流程

用户在浏览器提交表单 -> FastAPI 对应路由接收请求 -> Python 代码整理表单数据 -> 写入或更新 SQLite -> 查询最新任务列表 -> 页面重新渲染

## 数据库流程

新增任务时，后端把任务内容写入 SQLite；打开首页时，后端从 SQLite 查询任务列表；完成任务时，后端更新任务状态；删除任务时，后端从 SQLite 删除对应记录。

## 我学到的 3 件事

1. 先写 SDD 规格能让 Claude 更稳定。
2. 轻量 TDD 能帮助我先定义行为。
3. Review 阶段让我第一次真正看懂页面、路由和数据库之间的连接。

## 一个设计取舍

这个练习项目规模很小，所以把路由和数据库操作放在一起还能看懂。以后如果功能变多，可以考虑把数据库操作拆到单独文件里，让 main.py 更专注于请求和响应。

## 我看到的 2 个风险

1. 如果路由越来越多，main.py 可能会变得很长。
2. 如果表单输入没有检查，用户可能提交空任务。

## 下次我想改进的 2 件事

1. 给任务增加优先级
2. 把数据库操作再拆清楚一点

完成后检查：

cat docs/reflection.md
  1. 写完后，合上指南和 Claude 输出，自己尝试讲一遍“页面 -> 路由 -> SQLite -> 页面”的数据流。讲的时候可以按这 4 句话组织：
  • 用户在页面上做了什么？
  • 哪个路由接住了这个动作？
  • SQLite 在这一步被查询、写入、更新，还是删除？
  • 页面为什么能显示最新结果？

如果讲到一半卡住，就回到代码和 Claude 解释里找证据，再把 docs/reflection.md 改成更准确的说法。

输入输出对照

输入 预期输出 学生要比较什么
已完成项目文件 一份中文解释，说明关键文件、请求流程和数据库流程 是否能看懂项目为什么能工作
Claude 的解释和轻量 review 3 到 5 条可理解的改进建议 是否能把建议转成自己的判断，而不是原样复制
docs/reflection.md 包含项目结构、请求流程、数据库流程、学习点、取舍、风险和改进点 是否真的完成了复盘，而不是只写感想

验收标准

运行或检查：

cat docs/reflection.md

通过标准：

  • docs/reflection.md 已创建，并且包含项目结构、请求流程、数据库流程、学习点、取舍、风险和改进点。
  • 你已经拿到一份轻量 review 清单，并理解其中至少 2 条建议。
  • 你能区分“项目能跑”和“我能讲清楚它为什么能跑”。

需要停下处理：

  • 如果复盘还是流水账，没有解释项目结构，先回到关键文件和数据流。
  • 如果你只让 Claude 给建议，但自己不整理成文档，先把复盘改成自己的表达。
  • 如果你能看懂单个文件，却讲不出整体数据流，先回到代码和 Claude 解释里补证据。

卡点与过关

常见卡点：

  • 把复盘写成流水账，没有解释项目结构。
  • 只让 Claude 给建议，但自己不整理成文档。
  • 能看懂单个文件，却讲不出整体数据流。
  • 把“请求流程”和“数据库流程”混在一起。请求流程关注浏览器和路由怎样交互；数据库流程关注什么时候读、写、更新、删除数据。
  • 把 Claude 的解释当成作业答案。Claude 可以当助教，但最后的复盘要像你自己在向别人讲项目。
  • 只写优点，不写风险。Review 的意义不是批评自己，而是训练你看见当前实现的边界。

过关前确认：

  • docs/reflection.md 已创建。
  • 我已经能解释关键文件和数据流。
  • 我已经拿到一份轻量 review 清单，并理解其中至少 2 条建议。
  • 我能区分“项目能跑”和“我能讲清楚它为什么能跑”。
