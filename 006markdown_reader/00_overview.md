# 00. 开始之前：先建立 Vibecoding 的学习地图

## 预计用时

10-15 分钟。

## 本节位置

这是约 4 小时核心路线（00-10）的开场课。本节只负责建立概念地图和学习边界，不写代码、不创建项目、不启动 `Claude Code CLI`。你可以把它理解成上课前的“导航页”：先知道接下来会遇到哪些概念、这些概念为什么重要、它们分别会在后面的哪一节真正练习。

## 工具使用规则

本节不使用终端，也不启动 `Claude Code CLI`。你只需要阅读、理解和自查概念。全课统一工具边界见 [AGENT.md](../AGENT.md)。

## 学习目标

学完本节，你需要理解 5 个核心概念：`Vibecoding`、`Claude Code CLI`、`SDD`、`TDD`、`Review`。它们不是 5 个孤立的术语，而是一套互相配合的学习方式。

- `Vibecoding` 解决的是“怎样用自然语言和 AI 协作开发”的问题。
- `Claude Code CLI` 解决的是“怎样在真实项目目录里和 AI 一起工作”的问题。
- `SDD` 解决的是“怎样先把需求说清楚”的问题。
- `TDD` 解决的是“怎样判断代码行为是否符合预期”的问题。
- `Review` 解决的是“怎样从能跑走向能理解”的问题。

本节结束时，你不需要会写任何项目代码，也不需要记住复杂定义。你只需要能用自己的话回答 3 个问题：

1. 我接下来学习的 `Vibecoding` 到底是什么？
2. 为什么不能只把 AI 当成代写工具？
3. 后面的 `SDD`、`TDD`、`Review` 分别会帮我避免什么问题？

## 本节产出物

第一个产出物是一张概念对照表。你要能写清楚 5 个概念分别是什么意思、解决什么问题、后面在哪一节练习。

## 操作步骤

1. 先阅读“为什么要先建立地图”，理解本节为什么不急着实践。
2. 阅读 5 个核心概念，用自己的话复述每个概念。然后整理自己的概念卡，下面只是参考格式，不要求完全一致：

```markdown
# 我的 Vibecoding 概念卡

Vibecoding：
Claude Code CLI：
SDD：
TDD：
Review：
```

3. 阅读“安全 Vibecoding 工作流”，理解后面课程为什么采用 `想法 -> 规格 -> 任务 -> 实现 -> 验证 -> Review`。
4. 阅读“概念会在哪一节练习”，确认每个概念的实践位置。
5. 完成最后的检查清单，再进入 `01_vibecoding_modes.md`。本节不要求运行 `Claude Code CLI`；如果你已经熟悉 Claude，可以选择在 `Claude Code CLI` 中输入下面提示词帮助理解概念：

```text
请用 Python 新人能理解的语言解释 Vibecoding、Claude Code CLI、SDD、TDD 和 Review。不要生成代码，不要让我创建项目目录。
```

### 为什么要先建立地图

`Vibecoding` 的吸引力很强，因为它让人感觉“只要说出想法，AI 就能帮我做出软件”。Google Cloud 对 vibe coding 的解释也强调了这一点：它是一种用自然语言提示让 AI 生成、细化、调试代码的开发方式，能降低应用构建门槛，尤其对编程经验较少的人有帮助。

但对新人来说，降低门槛不等于取消学习。AI 可以帮你生成代码，却不能自动保证你理解代码。AI 可以根据你的描述做出一个原型，却不能替你判断需求是否合理。AI 可以修复某个报错，却不能替你建立长期的调试能力。你真正要学的不是“让 AI 写得更多”，而是“让自己更会指挥、更会判断、更会验证”。

这就是本课程采用 `SDD + TDD + Review` 的原因。它们像三道护栏，让 `Vibecoding` 不只是凭感觉往前冲。

`SDD` 让你先写清楚规格，避免一上来就让 AI 发挥过度。GitHub Spec Kit 对 Spec-Driven Development 的定位也很接近这个思路：在 AI 辅助开发中，把规格放在中心位置，让实现围绕规格展开。

`TDD` 让你先定义预期行为，再写实现。Martin Fowler 对 TDD 的说明里提到，TDD 的核心是通过测试来引导开发，常见节奏是 Red、Green、Refactor：先看到失败，再写最小实现让它通过，最后整理代码。

`Review` 则提醒你：代码能跑只是中间状态，不是学习终点。你还要能解释文件结构、数据流、关键函数，以及为什么这样设计。对新人来说，Review 不是“高级工程师才需要的步骤”，而是把 AI 生成结果变成自己知识的关键环节。

### 5 个核心概念

| 概念 | 一句话理解 | 它解决什么问题 | 新人最容易误解的地方 |
|---|---|---|---|
| `Vibecoding` | 用自然语言和 AI 协作开发，但由你负责目标、边界、验证和理解。 | 降低启动项目的门槛，让新人更早进入真实开发场景。 | 以为只要说一句“帮我做个系统”，剩下都不用管。 |
| `Claude Code CLI` | 在项目目录里工作的 AI 编程助手，可以帮助阅读文件、生成代码、解释报错和辅助修改。 | 把 AI 协作从聊天窗口带进真实代码环境。 | 以为它是自动完成整个项目的按钮，而不是需要你持续给上下文和验证标准的助手。 |
| `SDD` | 先写规格，再写代码。 | 防止需求模糊、范围失控、AI 自行扩展功能。 | 以为写规格是浪费时间，想直接进入代码。 |
| `TDD` | 先定义测试或预期行为，再写实现。 | 防止代码看起来能跑，但其实不符合需求。 | 以为 TDD 是写完代码后随便测一下。 |
| `Review` | 做完后解释、检查、复盘代码。 | 防止只会复制代码，不理解项目为什么能工作。 | 以为 Review 是可有可无的收尾，而不是学习发生的地方。 |

### 本课程采用的 Vibecoding 工作流

本课程不会采用“直接让 AI 生成完整项目”的方式。我们会采用更适合新人的工作流：

```text
想法 -> 规格 -> 任务拆分 -> 小步实现 -> 测试验证 -> Review 复盘
```

第一步是想法。你可以从很自然的话开始，比如“我想做一个个人任务管理系统”。这一步允许模糊，因为任何项目一开始都不会完美清楚。

第二步是规格。我们会把模糊想法变成更清晰的说明：系统给谁用、要解决什么问题、最小功能是什么、不做什么。这个阶段对应 `SDD`，它会在 `03_project_spec_sdd.md` 里真正实践。

第三步是任务拆分。一个完整系统对新人来说太大，对 AI 来说也容易发散。我们要把它拆成可以逐步完成的小任务，比如先创建后端入口，再接数据库，再做接口，再做页面。这个阶段会在 `04_task_breakdown.md` 里练习。

第四步是小步实现。每次只做一个目标，不要同时改 5 个地方。比如“只实现任务新增接口”，而不是“请把整个后端都做好”。这样一旦出错，你知道问题大概率来自哪一步。

第五步是测试验证。验证不是随便看一眼，而是有输入、有预期输出、有成功/失败判断。比如访问某个接口，应该返回什么 JSON；点击某个按钮，页面应该出现什么变化。`TDD` 会在 `01` 里用小函数先体验一次，在 `07` 里用于核心 API。

第六步是 Review 复盘。你会让 `Claude Code CLI` 帮你解释项目结构和数据流，但最后必须由你自己复述。你要知道页面如何提交表单，后端如何接收请求，SQLite 如何保存任务，页面如何重新显示结果。这个阶段会在 `10_review_and_reflection.md` 里完成。

### 对 Python 新人的特别提醒

如果你刚学 Python，不需要先把所有知识都学完再开始这门课。你只需要知道最基础的概念：变量、函数、列表、字典、`if`、`for`、虚拟环境、`pip`。后面遇到不懂的代码，我们会通过 `Claude Code CLI` 解释，通过输入输出对照验证，通过 Review 复盘。

但你也不能完全放弃理解。`Vibecoding` 对新人最有价值的地方，不是让你绕过学习，而是让你在真实项目里学习。以前你可能要学很久语法才敢做项目；现在你可以更早进入项目，但每一步都要问自己：我知道目标吗？我知道输入输出吗？我知道怎么判断成功吗？我能解释关键代码吗？

这四个问题会贯穿整套课程。

### 你在这门课里的三个身份

为了避免把 `Vibecoding` 学偏，你可以把自己想象成同时拥有三个身份：产品负责人、测试者、学习者。

第一个身份是产品负责人。哪怕这个项目很小，你也要负责回答“我们到底要做什么”。AI 很擅长根据提示生成内容，但如果你的目标含糊，它就会根据自己的默认理解补全细节。比如你只说“做一个任务管理系统”，AI 可能会加登录、用户权限、复杂前端框架、部署配置，甚至引入你还没准备学习的技术。产品负责人的工作，就是把范围收住：这次只做个人任务列表，只做新增、完成、删除，只用本地 SQLite，只追求能学习全栈链路。

第二个身份是测试者。测试者关心的不是“代码看起来漂亮不漂亮”，而是“输入之后输出是否符合预期”。你后面会不断看到“输入输出对照”和“成功/失败判断”，这不是形式主义，而是新人最需要建立的判断能力。没有判断标准，AI 每改一次你都只能凭感觉说“好像可以”。有了判断标准，你就能说：我访问这个地址，应该看到这个结果；我提交这个表单，数据库应该多一条记录；我点击删除，页面刷新后这条任务应该消失。

第三个身份是学习者。学习者的目标不是最快把项目做完，而是把每一步变成自己的理解。AI 可能一分钟生成几十行代码，但你不需要立刻理解每一个语法细节。你要先抓住主线：哪个文件负责启动应用，哪个函数负责处理请求，数据库保存了什么字段，页面上的按钮会触发哪个路由。等主线清楚了，再逐步理解细节。学习者最重要的问题不是“这段代码能不能再高级一点”，而是“我能不能向别人讲清它为什么存在”。

这三个身份会互相制衡。产品负责人防止需求发散，测试者防止结果失真，学习者防止只复制不理解。`Vibecoding` 真正适合新人的地方，就在于 AI 可以帮你加速，但这三个身份仍然由你来承担。

从 `01` 开始，每一节都会用一个简短的“三个身份提醒”把这三个身份重新拉回到当前任务里。

### AI 能帮你什么，不能替你什么

在这套课程里，`Claude Code CLI` 会做很多事。它可以帮你把模糊想法整理成规格，可以帮你拆任务，可以生成第一版代码，可以解释报错，也可以做轻量 Review。它像一个很有耐心的搭档，能让你少卡在“完全不知道从哪开始”的阶段。

但它不能替你做三件事。

第一，它不能替你决定学习目标。AI 不知道你现在是 Python 新人，也不知道你今天打算先完成约 4 小时的核心路线，除非你明确告诉它。所以后面每次提问，我们都会强调上下文：我是 Python 新人、我要做最小版本、请不要引入复杂技术。

第二，它不能替你承担最终判断。AI 给出的代码可能是合理的，也可能只是“看起来合理”。你必须通过运行、输入输出对照、测试、页面检查来确认结果。尤其是当项目变大时，AI 很可能修好一个地方又影响另一个地方，所以验证永远不能省。

第三，它不能替你形成长期能力。如果你只是复制 AI 的代码，今天也许能完成项目，明天换一个需求还是会慌。但如果你学会了先写规格、拆任务、定义预期、验证结果、复盘结构，你就能把这套方法迁移到博客系统、记账系统、打卡系统，甚至更复杂的项目里。

## 验收标准

运行或检查：

- 本节不用运行终端命令。只检查你是否已经完成概念卡，并能用自己的话解释本节 5 个核心概念。

通过标准：

- 你能说清 `Vibecoding` 是人主导、AI 协作，而不是 AI 全自动代写。
- 你能说清 `Claude Code CLI` 是在项目目录里工作的编程助手，不是普通聊天窗口。
- 你能区分 `SDD`、`TDD`、`Review`：一个先收清规格，一个先定义验证，一个帮助解释、检查和复盘。
- 你知道 `00` 只建立概念，不做项目；`01` 才做小函数方法实验；`02` 才创建正式项目目录 `task-vibe/`。
- 你能说出这些概念后面会在哪些章节被练习，而不是只背下定义。

需要停下处理：

- 如果你还不能解释 `SDD`、`TDD`、`Review` 各自解决什么问题，先补概念卡，不要进入 `01`。
- 如果你想在本节直接写代码、启动 Claude 或创建项目，先停下来；本节的目标是建立学习地图。
- 如果你已经准备创建 `task-vibe/`，先等到 `02`，不要把正式项目提前放进概念章节。

## 常见卡点

- 把 `Vibecoding` 理解成 AI 全自动代写。
- 把 `Claude Code CLI` 理解成普通聊天窗口，而不是项目目录里的协作工具。
- 把 `SDD` 和 `TDD` 混在一起，不知道一个偏规格，一个偏验证。
- 忽略 `Review`，以为代码能跑就等于已经学会。
- 想跳过概念地图，直接进入创建项目和写代码。
