# 04. 把规格拆成可执行任务

## 预计用时

15-20 分钟。

## 本节位置

约 4 小时核心路线（00-10）的第 4 个学习单元前半段。上一节已经把想做的应用写成了 `docs/spec.md`，这一节不急着写代码，而是把规格拆成能逐步完成、逐步验证的小任务。

## 工具使用规则

终端用于确认和查看 `docs/spec.md`、保存和检查 `docs/tasks.md`。`Claude Code CLI` 用于把规格拆成小任务，并为每个任务补目标、涉及文件和验证方式。本节的重点不是让 AI 开始改代码，而是先让 AI 帮你写出一份清楚的施工清单。

## 学习目标

把 `docs/spec.md` 拆成一组适合 `Claude Code CLI` 执行的小任务，让每一步都足够小、足够清楚，而且做完后能立刻验证。

你要掌握的核心点是：任务拆解就是把一份规格变成小而可验证的工作单元。对 Python 新人来说，这比“让 AI 直接做完整项目”更稳，因为你可以一次只理解一个变化、检查一个结果、修正一个问题。

## 本节产出物

`task-vibe/docs/tasks.md`

## 操作步骤

1. 确认你还在 `task-vibe/` 目录内，并先完整查看规格。在终端执行：

```bash
cd task-vibe
pwd
cat docs/spec.md
```

读 `docs/spec.md` 时，把自己当成接下来要施工的人：不要只看应用名字，要找出“用户最终能做什么”和“为了做到这件事，代码里必须出现哪些部分”。如果规格里没有写登录、多用户、标签、搜索、部署，就先不要把这些内容加进任务里。

2. 提取最小交付链路。这个示例应用的技术范围不要扩大，只围绕规格里的最小链路整理：项目骨架、数据库、接口、页面、调试、复盘。你可以在心里先这样问：

- 应用最早能跑起来需要什么？
- 数据要保存在哪里？
- 用户新增任务需要哪个接口或表单？
- 用户查看任务需要哪个页面或返回结果？
- 做完后我能用什么命令、页面或请求确认它真的完成了？

3. 理解“一个任务只做一件事”。这里的一件事不是一句很短的话，而是一个可以独立完成、独立检查的变化。例如“创建最小 FastAPI 应用”是一件事，因为做完后可以立刻访问 `/` 看返回值；“完成后端”不是一件事，因为它可能包含数据库、接口、校验、错误处理等许多变化。

这条原则对 AI 协作特别重要。AI 在小任务里更容易保持目标清楚，也更不容易顺手添加规格之外的功能。你给它一个小任务，它就能围绕少量文件工作；你给它一个大任务，它可能同时改很多地方，出了问题你也很难判断是哪一步造成的。小任务还能让你每完成一步就验证一次，避免把十个问题累积到最后才发现。

4. 让 `Claude Code CLI` 按“一个任务只做一件事”的原则拆分任务。

Claude Code 提示词示例：

```text
请阅读 docs/spec.md，创建或更新 docs/tasks.md，把规格拆成 8 到 12 个适合 Python 新人完成的小任务。

要求：
1. 一个任务只做一件事。
2. 每个任务必须包含：目标、涉及文件、完成后如何验证。
3. 任务顺序要能从空项目逐步做到可用应用。
4. 不要加入 docs/spec.md 之外的新功能。
5. 如果你觉得某个功能值得以后做，请放到“暂不实现”里，不要放进本次任务。
```

5. 检查 Claude 给出的任务是否太大。一个任务通常太大，如果它出现这些信号：

- 任务名里有“完成整个”“实现所有”“优化系统”“处理后端逻辑”这类笼统词。
- 一步同时要求改数据库、接口、页面和样式。
- 涉及文件多到你还没开始就说不清每个文件为什么要改。
- 验证方式只能写“看起来正常”或“运行看看”，没有具体结果。
- 你估计一个 Python 新人需要超过一轮专注练习才能完成它。

遇到大任务时，不要勉强执行。把它拆成更小的任务，例如先“创建 tasks 表”，再“添加新增任务接口”，再“在页面表单里调用新增接口”。

6. 删除规格漂移。规格漂移就是任务清单里出现了 `docs/spec.md` 没要求的东西。对新人来说，这会把练习从“完成一个小应用”变成“追逐一堆新想法”。如果 Claude 输出了登录、用户头像、标签系统、复杂筛选、云部署、AI 自动分类等额外内容，先删掉，或放进“暂不实现”。本节只保留能完成样例应用最小范围的任务。

7. 要求每个任务都写出涉及文件和完成后的验证方式。涉及文件让 AI 知道主要工作边界，也让你知道该检查哪里；验证方式让任务变成可验收的工作，而不是一句愿望。没有目标文件，AI 容易到处改；没有验证方式，你就不知道任务什么时候算完成。

好的任务通常长这样：

```markdown
- [ ] 创建最小 FastAPI 应用
  - 目标：让项目能启动，并提供一个健康检查入口。
  - 涉及文件：main.py
  - 验证方式：运行应用后访问 `/`，返回 `{"message":"Task Vibe is running"}`
```

含糊任务通常长这样：

```markdown
- [ ] 完成后端
  - 涉及文件：很多文件
  - 验证方式：运行看看
```

第一个任务能执行、能检查、能判断对错。第二个任务太大，既没有清楚边界，也没有具体的成功信号。

8. 让 Claude Code 将收缩后的结果保存为 `docs/tasks.md`。下面的任务片段只用来对照结构，不要求完全一致。

参考修改示例：

```markdown
- [ ] 创建最小 FastAPI 应用
  - 目标：让项目能启动，并提供一个健康检查入口
  - 涉及文件：main.py
  - 验证方式：访问 `/` 返回 `{"message":"Task Vibe is running"}`

- [ ] 创建 SQLite tasks 表
  - 目标：准备保存任务数据的表结构
  - 涉及文件：db.py 或 main.py
  - 验证方式：数据库文件存在，tasks 表可以查询

- [ ] 添加新增任务接口
  - 目标：允许用户提交一个新的任务标题
  - 涉及文件：main.py
  - 验证方式：提交标题后，任务能出现在列表里
```

完成后检查：

```bash
cat docs/tasks.md
```

如果你看到 8 到 12 个 checkbox 任务，而且每个任务下面都有“涉及文件”和“验证方式”，就可以进入本节最后的检查。如果少了这些字段，回到 Claude Code CLI，让它按同一格式补齐，不要直接进入下一节。

## 输入输出对照

| 输入 | 预期输出 | 学生要比较什么 |
|---|---|---|
| `docs/spec.md` | `docs/tasks.md`，包含 8 到 12 个 checkbox 任务 | 是否每个任务都来自规格，而不是来自 Claude 的额外想象 |
| “完成后端” | “创建 SQLite tasks 表”“添加新增任务接口”“添加查询任务接口” | 是否把大块工作拆成了可独立验证的小块 |
| “做好页面” | “创建任务列表页面”“添加新增任务表单”“提交后刷新列表” | 是否能说清每一步改哪个文件、看什么结果 |
| “优化一下” | 暂不放入本次任务，除非 `docs/spec.md` 明确要求 | 是否及时剪掉范围漂移 |
| 每个任务的文字 | 目标、涉及文件、验证方式 | 是否回答了做什么、改哪里、怎么验收 |

## 验收标准

运行或检查：

```bash
cat docs/tasks.md
```

通过标准：

- 任务数量在 8 到 12 个之间。
- 每个任务都能在一轮专注工作里完成，并且有明确验证方式。
- 你能从第一条一路读到最后一条，感觉它们像一条施工路径，而不是一堆愿望。

需要停下处理：

- 如果任务太大，例如“完成后端逻辑”这种无法一步验证的表述，继续拆细。
- 如果任务没有写涉及文件和验证方式，先补完整再继续。
- 如果任务清单里出现了规格之外的新功能，比如登录、权限、部署或复杂筛选，直接删掉或移到“暂不实现”。

## 卡点与过关

常见卡点：

- 把“页面、接口、数据库”揉成一个任务。
- 验证方式只写“看起来可以”，不写具体结果。
- 让 Claude 一次生成太多任务，导致自己看不懂顺序。
- 任务里没有目标文件，导致 AI 不知道主要应该改哪里。
- 看到 Claude 给了更多功能就全部接受，结果任务清单越来越大。
- 不先读 `docs/spec.md`，直接让 Claude 发挥，最后得到的任务和原规格对不上。

过关前确认：

- 我已经创建了 `docs/tasks.md`。
- 每个任务都包含目标、涉及文件、验证方式。
- 我能说出下一节先做哪一个任务，以及为什么。
- 我已经删掉或移出了 `docs/spec.md` 没要求的新功能。
- 我已经用 `cat docs/tasks.md` 检查过保存后的任务清单。
