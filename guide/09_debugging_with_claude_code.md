# 09. 出现错误时，怎样和 Claude Code CLI 一起调试

## 预计用时

20-25 分钟。

## 本节位置

约 4 小时核心路线（00-10）的第 7 个学习单元后半段。用真实报错或受控练习错误，练习“先定位根因，再做最小修复”。

## 工具使用规则

终端和浏览器用于确认 `task-vibe/.venv`、复现错误、收集完整输出、重新验证同一个动作。`Claude Code CLI` 用于分析报错、定位根因、给出最小修复方案。

这一节的核心不是“看到错误就让 AI 改到不报错”，而是练习一条稳定的调试循环：先稳定复现，再收集证据，然后隔离最可能的原因，只做最小修复，最后用完全相同的动作重新验证。这个顺序很重要。Python 新手最容易卡在“我觉得是这里错了”，但真实项目里，感觉经常会被浏览器缓存、旧服务进程、拼错的 URL、表单 method、路由路径、字段名等细节误导。调试记录就是用来把这些细节固定下来，让你和 `Claude Code CLI` 都基于证据说话。

使用 `Claude Code CLI` 时，把它当成分析伙伴，不要一开始就让它“大范围帮我修好”。你可以先要求它只读、只分析、只列出最小修改建议；等你确认原因以后，再允许它改动明确的一个或两个文件。这样做可以避免它顺手重构模板、格式化无关文件、改掉你还没学过的结构，导致原本的小错误变成一堆新问题。

## 学习目标

- 学会把报错和上下文一起交给 `Claude Code CLI`，而不是只说“帮我修一下”。
- 学会要求它先分析原因，再给最小修复方案。
- 学会判断一个问题是不是真的修好了，而不是“好像没报错了”。
- 学会区分“错误现象”“证据”“猜测”和“根因”。
- 学会用同一个浏览器动作或终端命令验证修复，而不是换一种方式绕开问题。

## 本节产出物

- 当前工作目录：`task-vibe/`
- 一份调试记录，可以保存在 `docs/debug-note.md`
- 至少修掉一个真实问题，或者完成一轮受控调试练习

重点看三件事：有没有原始错误、有没有最小修复、有没有用同一个动作重新验证。

如果你在第 08 节已经把页面做通了，这是好消息，不是问题。第 09 节不要求你等一个真实 bug 自己出现；如果当前项目已经能正常新增、完成、删除任务，就用一个明确、可恢复的小错误来练习调试流程。

一份合格的调试记录不需要写得很长，但必须能让别人复现你的问题。别人读完以后，应该知道你在哪个目录、访问哪个 URL、点了哪个按钮、请求了哪个 route、终端显示了什么、浏览器实际发生了什么、你原本期待发生什么、最后改了哪里，以及为什么这个改动足够小。

## 操作步骤

1. 继续在 `task-vibe/` 目录里工作。
2. 先确认目录和虚拟环境，再确认你能启动当前小应用。在终端执行：

```bash
cd task-vibe
source .venv/bin/activate
python -m pip show fastapi uvicorn httpx2 python-multipart
python -m uvicorn main:app --reload
```

使用前面课程里已经用过的启动方式，不要为了调试临时换框架、换目录、换端口。启动后保留终端窗口，因为 Python traceback、HTTP 状态码、请求路径通常会出现在这里。读文件、记录浏览器 URL、使用 `curl` 不一定需要虚拟环境；但只要你要启动服务、运行测试或执行 Python 片段，就先确认 `task-vibe/.venv` 已激活。

3. 先确认第 08 节的工作版本确实正常。打开 `http://127.0.0.1:8000/`，按顺序做三件事：

- 新增一条任务。
- 点击一次完成切换。
- 删除刚才那条任务。

如果三件事都正常，说明你的第 08 节已经完成得不错。接下来走“受控练习错误”路径：人为制造一个很小、可解释、可恢复的错误来练习调试。不要为了找 bug 随便乱改多个文件。

如果这三件事里已经有一个失败，就走“真实错误”路径：不要再制造新错误，直接用这个真实问题练习后面的调试步骤。

4. 如果当前没有真实错误，就制造一个受控练习错误。推荐默认错误是：只把 `templates/index.html` 里新增任务表单的 `action="/tasks"` 临时改成 `action="/"`。这样点击新增任务时，浏览器会用 `POST /` 提交表单，常见结果是 `405 Method Not Allowed`。这个错误很小，和表单提交地址、后端路由方法直接相关，调试完也容易恢复。

这个受控错误的原值是 `action="/tasks"`，练习结束必须改回去。你要记住：第 09 节练的是调试流程，不是把项目长期弄坏。

5. 稳定复现一次错误。复现时只做一个清楚的动作```

```text
打开 http://127.0.0.1:8000/
在任务输入框输入“学习 Claude Code”
点击新增按钮
观察页面和终端输出
```


复现时不要连续乱点，也不要一边改代码一边猜。你现在要得到的是“同一个动作，每次都会触发同一个问题”。如果同一个动作不能稳定复现，先把不稳定现象也记录下来，例如“第一次返回 405，刷新后变成 404”，这本身就是重要证据。

如果你走的是受控练习错误路径，预期现象通常是：输入任务标题并点击新增后，页面出现 `405 Method Not Allowed`，或者终端里能看到和 `POST /` / 方法不匹配有关的记录。

6. 收集证据，而不是只收集一句报错。至少记录下面六类信息：

- 终端输出：从你执行命令开始，到错误结束为止的完整输出。Python traceback 不要只截最后一行。
- 浏览器动作：你打开了哪个页面、输入了什么、点击了哪个按钮。
- URL 和 route：浏览器地址栏里的 URL，以及终端里出现的请求路径，例如 `GET /`、`POST /tasks`。
- 相关文件片段：只贴和问题有关的短片段，例如 `main.py` 里定义 route 的几行、`templates/index.html` 里 `<form>` 的几行。
- 预期行为：你本来期待它做什么，例如“提交后新增一条任务并回到首页”。
- 实际行为：它实际做了什么，例如“页面显示 405 Method Not Allowed，任务没有新增”。

你可以用这些只读命令帮助自己定位上下文：

```bash
pwd
ls
cat main.py
cat templates/index.html
```

读文件时先看小范围。新手常见问题通常就在 route、form、变量名、模板文件名附近，不需要一开始把整个项目都贴给 `Claude Code CLI`。

7. 让 Claude Code 根据你的证据创建或更新 `docs/debug-note.md`，记录触发动作和完整报错。下面是参考模板，不要求完全一致，但必须保留能复现问题的证据。

9. 用同一个动作重新验证。也就是说，修复前如果是“打开首页 -> 输入文字 -> 点击新增按钮”触发错误，修复后就必须再次执行完全相同的动作。不要换成直接访问别的 URL，也不要只看服务能不能启动。只有同一个动作通过，才能说明原来那条失败路径被修好了。

验证时继续记录三件事：

- 终端是否还出现同样的 traceback、状态码或 route 错误。
- 浏览器是否完成了原本预期的行为。
- `docs/debug-note.md` 里的“验证结果”是否写清楚“同一动作已通过”。

10. 验证本节结果。

修复后，必须重复执行完全相同的动作，而不是换一种方式试试看。只有同样的动作不再报错，才算这次调试真正完成。

原因是：一个 bug 往往绑定在一条具体路径上。比如受控练习里的 `POST /` 报 405，或者真实错误里的 `POST /tasks` 报 405，都说明“表单提交到这个 route”这条路径失败；如果修复后你只访问 `GET /`，只能证明首页能打开，不能证明表单提交已经修好。验收标准必须是“同一动作通过”，因为它重新测试了原来失败的入口、请求方法、URL、后端 route、字段读取和页面跳转。换动作可能只是绕开了 bug，而不是修掉了 bug。

如果修复后出现了新的错误，也不要立刻把它和原错误混在一起。先判断它是不是同一个动作继续触发的下一层问题。如果是，把新的终端输出和浏览器表现追加到同一份调试记录；如果不是，另开一份调试记录，避免多个问题互相污染。

如果你走的是受控练习错误路径，最后还要重新做一次第 08 节的三步动作：新增、完成、删除。三步都通过，才说明你已经把项目恢复到工作版本。


## 输入输出对照

| 输入 | 预期输出 | 学生要比较什么 |
|---|---|---|
| 浏览器点击新增按钮后出现错误页 | 一条带原因分析和修复方案的调试记录 | 是否把“错误 + 触发动作 + URL/route + 预期/实际 + 相关文件片段”一起交给了 Claude |
| 终端出现 Python traceback 或 HTTP 状态码 | 调试记录里保留完整输出，而不是只保留最后一行 | 是否能从证据看出错误发生在哪个请求或哪一行 |
| Claude 给出修改建议 | 只围绕最可能根因的最小修复方案 | 是否拒绝了无关重构、无关格式化和新增功能 |
| 修复后再次执行相同动作 | 不再报错，页面行为恢复正常 | 是否真的复现并验证了修复，而不是换了另一条路径 |

## 验收标准

运行或检查：

```bash
cat docs/debug-note.md
git diff
```

通过标准：

- 你已经完成一份调试记录，而且证据包含错误、触发动作、URL/route、预期/实际和相关文件片段。
- 你已经让同一个失败动作重新执行并验证过。
- 修复后页面行为恢复正常，`git diff` 里没有无关重构、无关格式化和新增功能。

需要停下处理：

- 如果你只贴一行报错，没有贴触发动作，先补证据，不要急着修。
- 如果 Claude 一次性修一堆问题，先收回到最可能根因的最小修复方案。
- 如果修完以后没有重复执行同一动作，先不要进入下一节。

## 卡点与过关

常见卡点：

- 只贴一行报错，没有贴触发动作。
- 让 `Claude Code CLI` 一次性修一堆问题，结果改动失控。
- 修完以后没有重复执行同一动作。
- 只记录“页面报错”，没有记录浏览器 URL、请求 route 和终端输出。
- 把“我猜是表单错了”写成结论，但没有贴出表单和 route 的代码片段。
- 看到 Claude 给出很多建议就全部接受，忘了本节目标是最小修复。
- 修复后只重启服务，没有重新在浏览器里执行原来的点击动作。

过关前确认：

- 我已经完成一份调试记录。
- 我已经让同一个失败动作重新执行并验证过。
- 我能说出这次错误的根因，而不是只知道“现在不报错了”。
- 我能指出这次修复改了哪个文件、哪几行，以及为什么没有改其他文件。
